# 01 项目目标与 MVP 范围

## 1. 项目背景

本项目面向 AIGC 个人作品集中的“情景与剧情生成”方向。

它不是简单做一个“AI 写故事工具”，而是尝试建立一个具有“世界状态演化”思想的叙事生成框架。

普通故事生成工具常见问题：

- 前后剧情不一致。
- 人物动机突然变化。
- 事件之间缺乏因果关系。
- 场景生成和剧情发展脱节。
- 图像 Prompt 只描述画面，但不服务于剧情推进。

本项目希望解决的核心问题是：

> 如何让剧情生成不只是连续写作，而是像一个“世界”在不断演化？

---

## 2. 项目定位

项目暂定名：**StoryWorld**

完整定位：

> StoryWorld 是一个基于结构化世界状态的情景—剧情联合生成系统。系统通过维护人物、事件、线索、冲突和场景状态，使 LLM 生成的剧情能够保持较强的连续性和可视化转化能力。

简单说：

```text
不是：一句话 → 一段故事
而是：设定 → 世界状态 → 事件推进 → 状态更新 → 场景生成 → 图像 Prompt
```

---

## 3. MVP 输入

用户输入一段自然语言故事设定，例如：

```text
我想生成一个校园悬疑故事，主角是计算机专业女生，故事发生在深夜宿舍，她收到一封来自未来的邮件。
```

---

## 4. MVP 输出

第一版系统输出：

1. 结构化故事设定 `StorySetting`
2. 初始世界状态 `WorldState`
3. 3～5 个剧情事件 `EventCard`
4. 每个事件之后更新的世界状态
5. 每个事件对应的场景卡 `SceneCard`
6. 每个场景对应的图像 Prompt

---

## 5. MVP 不做什么

第一版不要做：

- 不从零训练 LLM。
- 不从零训练 Diffusion。
- 不做复杂强化学习。
- 不做真正具身智能。
- 不做大型多人剧情系统。
- 不做商业级游戏编辑器。
- 不做复杂数据库系统。

第一版要做的是：

> 把“世界状态约束剧情生成”的核心链路跑通。

---

## 6. 项目核心创新点

### 6.1 世界状态驱动剧情

系统维护一个 `WorldState`，记录当前时间、地点、人物状态、已知线索、未解决冲突等内容。每次剧情生成时，不是凭空生成，而是基于当前 `WorldState` 推进下一步事件。

### 6.2 事件反向更新世界

每一个剧情事件都会带来状态变化：

- 人物情绪变化
- 目标变化
- 关系变化
- 新线索出现
- 冲突升级或解决
- 场景发生变化

这使得系统具有“演化感”。

### 6.3 场景卡连接文本与图像

系统不会直接把剧情丢给图像模型，而是先生成 `SceneCard`。

`SceneCard` 记录：

- 地点
- 时间
- 人物动作
- 镜头语言
- 光照
- 氛围
- 关键物品
- 画面 Prompt

这样可以让剧情与情景图像之间有明确的中间层。

---

## 7. MVP 成功标准

第一版成功标准不是“生成结果多惊艳”，而是满足以下条件：

1. 输入任意故事设定，程序能完整跑通。
2. 输出内容是结构化 JSON + 可读文本。
3. 连续生成 3 个以上事件时，基本不自相矛盾。
4. 每个事件都能导致世界状态变化。
5. 每个事件都能转成场景卡。
6. 每个场景卡都能转成图像 Prompt。
7. 结果可以保存为 Markdown 或 JSON 文件。
